# 时间胶囊：此地有银三百两

一个 Sepolia 测试网网页钱包原型：**一个用户 = 一个钱包；一次充值/埋银子 = 一张时间订单**。

## 核心设定

- 不导入助记词；
- 用户只有一个 Token Core WASM 钱包；
- 第一笔埋银子时创建唯一钱包，之后一直使用同一个充值地址；
- 用户只看到唯一充值地址、订单列表和打码后的转出地址；
- 每张订单必须设置：
  - 埋多少 ETH；
  - 多久后自动转出；
  - 到哪个地址；
- 到时间后，打开网页自动尝试从同一个钱包按订单金额转到目标地址；
- 如果失败，记录失败，下次打开继续尝试；
- 不用合约，第一版只验证「时间胶囊钱包」体验。

## 为什么“埋银子”不是新建钱包

产品上用户应该只有一个钱包，否则体验会变成“批量开地址工具”。

正确逻辑是：

```text
唯一钱包：0xABC...
  订单 A：0.1 ETH，1 年后转给地址 X
  订单 B：0.5 ETH，10 年后转给地址 Y
  订单 C：0.02 ETH，30 天后转给地址 Z
```

用户每次想埋一笔银子，就创建一张订单，然后往同一个钱包地址充值对应金额。

## 无合约版本的重要边界

ETH 是账户模型，同一个钱包里的余额天然是混在一起的。没有合约时，链上不会知道哪一笔充值属于哪张订单。

所以本 demo 的订单是“产品账本”：

- 页面记录订单金额、时间、目标地址；
- 到期后页面用同一个钱包签名转账；
- 如果余额不足，就下次打开继续检查；
- 这不是链上强制锁仓。

如果未来要做到每笔资金链上隔离，才需要合约或为每笔订单创建单独地址。但当前用户体验方向优先采用“一个钱包”。

## Passkey 能不能存助记词？

不应该把助记词“存进 Passkey”。Passkey 的正确作用是：

1. 证明是本人在操作；
2. 在支持 WebAuthn PRF / hmac-secret 的环境中，派生一个稳定的解密密钥；
3. 用这个密钥解密备份文件中的唯一钱包 keystore 和订单信息。

所以正式设计应该是：

```text
Passkey 解密密钥 + 加密备份文件 = 恢复唯一钱包和所有订单
```

当前 demo 用 Passkey 做本人确认；生产版应升级为 Passkey PRF 加密/解密。

## 备份与去中心化

用户可以选择不导出备份：当前浏览器仍能正常工作。

用户也可以导出备份：备份应该包含加密后的：

- 唯一钱包 keystore；
- 订单列表；
- 每张订单的转出时间；
- 打码前的目标地址；
- 必要的恢复元数据。

为了真正去中心化，除了备份 JSON，还应该把整个静态网页 `dist/` 一起导出。这样多年后即使网站没了，用户也可以打开离线网页，加载备份并执行到期转账。

生成离线网页包：

```bash
npm run portable
```

会输出：

```text
portable/time-capsule-silver-dist.zip
```
